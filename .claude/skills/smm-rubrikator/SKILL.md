# SMM-рубрикатор 3.1 (ЧиП)

## Цель

Рубрикатор превращает стратегию из SMM-концепции в конкретные посты: какой контент, в какой день, с какой целью. **Без рубрикатора концепция — набор слайдов.**

---

## Шаг 1: Выбрать 7 рубрик из карты коммуникаций

Алгоритм:
1. Из каждого этапа воронки — 1–2 сценария с наибольшим потенциалом
2. Проверить баланс: **40% привлечение / 25% прогрев / 20% продажа / 15% лояльность**
3. Оставшиеся сценарии — в резерв для ротации
4. Утвердить выбор с руководителем (для нас = Critic валидирует)

Правила выбора:

- **Привязка к дню НАВСЕГДА.** «Группа недели» в понедельник = каждый понедельник. На этом строится аналитика.
- **Уникальные названия.** НЕ «продающая», «экспертная» — это категории. Каждая рубрика с индивидуальностью.
- **5–7 рубрик на старте.** Меньше 5 — мало касаний. Больше 7 — перегрузка.
- **1 гибкий слот (воскресенье)** — инфоповод или плановая.
- **ID сценария из карты коммуникаций ОБЯЗАТЕЛЕН** в каждой рубрике.

Пошаговый алгоритм с примерами для Эстрим Феста: @references/rubric-selection-algorithm.md

## Шаг 2: Справочник рубрик в таблице

Справочник = блок из 7 строк в правой части листа «Контентная» (колонки AB–AF).
Содержит: название, ID, этап воронки, формат, маяк.
Контент-план подтягивает данные **АВТОМАТИЧЕСКИ через VLOOKUP** — не вводить одно и то же дважды.

**Маяк** = лучший пост рубрики.
- На старте маяк = лучший пост конкурента (Скрежет металла или смежные ниши).
- Через 2 недели — свой лучший пост этой рубрики.

**Грейд проверки:**
- Прогрев + Продажа (~45%): маркетолог/Critic проверяет **ДО** публикации
- Привлечение + Лояльность (~55%): ContentOps публикует сам, Critic смотрит выборочно

---

## Шаг 3: Двухнедельная проверка рубрик

**КЛЮЧЕВАЯ МЕХАНИКА.** Каждые 2 недели:

1. Открыть сводку по рубрикам (автоматически считается в таблице)
2. Определить лучшую (ранг 1–2, зелёная) и худшую (ранг 6–7, красная)
3. Для худшей — 3 сценария переработки:
   - **а) Изменить ПОДАЧУ** (заголовок, визуал) — не трогать тему
   - **б) Изменить МЕХАНИКУ** (опрос→сохрани, серия→карусель) — не трогать подачу
   - **в) ЗАМЕНИТЬ рубрику целиком** из резерва
4. **ОДНО изменение за раз.** Если менять всё сразу — не поймёшь что сработало.
5. Обновить маяки

**Когда менять рубрику:** МИНИМУМ 4 публикации (4 недели) до решения о замене. Алгоритмы раскачивают контент 2–3 недели. 2 поста — слишком мало для выводов.

## Баланс воронки (автоматический в таблице)

| Этап | % план | Постов/нед |
|------|--------|-----------|
| Привлечение | 40% | 2–3 |
| Прогрев | 25% | 1–2 |
| Продажа | 20% | 1–2 |
| Лояльность | 15% | 1 |

Формулы COUNTIF в таблице считают автоматически. **НЕ ТРОГАТЬ формулы.**

Перед созданием черновика: проверь текущий баланс по неделе. Если воронка перекошена — выбирай S-ID из недостающего этапа.

## Обязательная структура каждого поста (метаданные)

Каждый пост в контент-плане содержит:

```markdown
## МЕТА
- S-ID: [из карты коммуникаций, обязательно]
- Рубрика: [подтягивается VLOOKUP]
- Этап воронки: [подтягивается VLOOKUP]
- Маяк: [подтягивается VLOOKUP]
- Сегмент: [«Свои» / «Паломники» / «Скептики» / «Музыканты» / «Искатели нового»]
- Текст поста: [полный текст]
- Цель текста: [зачем этот пост — одно предложение]
- Продающая часть А→Б: [точка А зрителя → точка Б после прочтения]
- Как сделать интересным: [механика вовлечения из 8 ниже]
- Формат: [Reels / Пост / Карусель / Stories / VK Video]
- Охват план: [число]
- Охват факт: [заполняет Женя после публикации]
- ERR% план: [число]
- ERR% факт: [заполняет Женя после публикации]
- Грейд: [Critic до публикации / ContentOps сам, Critic выборочно]
```

**Продающая часть А→Б** — не CTA «купи билет». Это нарратив:
- **А** — текущая ситуация, боль, барьер аудитории
- **Б** — результат после взаимодействия с постом или фестом

Пример: А: «Слышал об этой группе, но не видел живьём» → Б: «11 июля — увижу. Осталось решиться»

## Механики вовлечения (8 штук, выбирать 1–2 на пост)

| Механика | Как работает | Пример для Эстрим Феста |
|----------|-------------|------------------------|
| **Опрос/тест** | 2–4 варианта стикером | «Какая группа круче? Master vs Воля и Разум» |
| **Верю/не верю** | Утверждение + голосование | «Эктоморф собрал 2500 на сольнике в Москве» |
| **Сохрани** | Полезная таблица/чеклист | «Чеклист: что взять на метал-фест» |
| **Серия 1/N** | История по частям | «Группа недели 1/12» |
| **+ в DM** | «Напиши + — пришлём программу» | «Хочешь полную программу? + в DM» |
| **До/после** | Визуальная трансформация | «Площадка до монтажа → готовая сцена» |
| **Цифра-удивляет** | Неожиданная статистика | «В России 1400+ метал-групп. 12 лучших — у нас» |
| **Закулисье** | Процесс, команда, детали | «Как мы выбирали звук для летней сцены» |

## Google Sheets

Работаем с **СУЩЕСТВУЮЩЕЙ** таблицей (`GOOGLE_SPREADSHEET_ID` из `.env`).

**Формулы VLOOKUP и COUNTIF НЕ ТРОГАТЬ.**

Агент пишет **ТОЛЬКО** в data-строки (строки 4+, колонки A–Q):
- Текст поста (H)
- Цель текста (I)
- Продающая часть А→Б (J)
- Как сделать интересным / механика (K)
- Формат (L)
- Охват план (M)
- ERR% план (O)
- Статус → «Черновик» (Q)

Колонки R+ (баланс воронки, справочники) — **запрет на запись**.
Команда синхронизации: `node scripts/migrate-drafts-to-sheet.js`

## Gotchas

- Каждый пост = одна идея. Не пытаться запихать всё в один пост
- Первые 3 строки = хук. Аудитория решает листать дальше за 2 секунды
- Контент-план готовится на 7 дней вперёд, не поштучно
- «Нейросетей здесь не было» — указывать там где уместно (скетчи Оли, ручной контент)
- Обратный отсчёт (14 дней до феста): минимум 4–5 форматов вариации, не один шаблон
- AI-арт/AI-изображения — НИКОГДА. Давид рисует сам
