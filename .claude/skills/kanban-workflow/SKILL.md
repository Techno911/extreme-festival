# Стандарт работы в Kanban 2.4 (ЧиП)

## 5 колонок доски

```
Бэклог → В работе → На проверке → Не принято → Готово
```

| Колонка | Что здесь | Кто работает |
|---------|----------|-------------|
| **Бэклог** | Задачи сформулированы, ждут исполнителя | Никто — задача ждёт |
| **В работе** | Задача взята в работу | Исполнитель (агент или Женя) |
| **На проверке** | Результат готов, ждёт утверждения | Женя или Critic |
| **Не принято** | Проверяющий вернул с комментарием | Исполнитель (доработать) |
| **Готово** | Утверждено. Закрыто. | — |

---

## Pull-система (ОБЯЗАТЕЛЬНО)

**Исполнитель сам берёт задачу из бэклога** — когда закрыл предыдущую.
Никто не назначает задачи принудительно.

**WIP-лимит: максимум 1 задача в работе на человека.**
Взял вторую — нарушение стандарта. Исключение: первая заблокирована (см. раздел «Блокеры»).

## SMART-постановка задачи

Каждая задача обязана быть SMART:

| Буква | Критерий | Проверочный вопрос |
|-------|---------|-------------------|
| **S** — Specific (конкретная) | Чёткий результат на выходе | «Что именно будет сделано?» |
| **M** — Measurable (измеримая) | Можно проверить без объяснений | «Как узнать, что готово?» |
| **A** — Achievable (достижимая) | Реально за одну неделю одним человеком | «Успеет один человек за 5 дней?» |
| **R** — Relevant (связана с целями) | Двигает к продаже билетов / тактике | «Зачем это нужно фесту?» |
| **T** — Time-bound (с дедлайном) | Конкретная дата. По умолчанию: **пятница 17:00** | «Когда точно?» |

**Примеры:**

❌ Плохо: «Написать пост про группу»
✅ Хорошо: «Черновик поста ВК (800–1000 зн.) + ТГ (200–300 зн.) про группу Master. Файл: `output/drafts/2026-04-18-gruppa-nedeli-master.md`. Дедлайн: 18.04.2026 пт 17:00. Ответственный: content-ops.»

❌ Плохо: «Исследовать конкурентов»
✅ Хорошо: «Таблица сравнения Скрежет vs Эстрим Фест: цены, лайнап, подписчики ВК, engagement. Источники: vk.com/screzhmet + TicketsCloud. Файл: `output/research/competitor-2026-04-18.md`. Дедлайн: 18.04.2026.»

## Обязательные поля карточки

```
Название:     [глагол + результат]
Ответственный: [агент или Женя Аржевский]
Дедлайн:      [дд.мм.гггг пт 17:00]
Результат:    [конкретный файл или артефакт]
Чек-лист:     [если 3+ действий — обязательно]
```

**Правило чек-листа:** если для выполнения задачи нужно 3 или больше отдельных действия — разбей их на пункты `- [ ]`. Без чек-листа задача с 3+ шагами не берётся в работу.

## Атомизация задач

**Если задача > 2 дней — дроби на подзадачи.**

Признаки что задачу нужно разбить:
- В описании больше двух глаголов-действий
- Есть зависимость от другой незакрытой задачи
- Нет чёткого файла на выходе

Каждая подзадача — самостоятельная карточка со своим SMART и своим исполнителем.

## Definition of Done (DoD) — переходы

Полный чеклист: @references/dod-checklist.md

### Бэклог → В работе
- [ ] Задача сформулирована по SMART
- [ ] Известен конкретный файл/артефакт на выходе
- [ ] Дедлайн проставлен
- [ ] Ответственный определён
- [ ] Если 3+ действий — чек-лист есть
- [ ] Предыдущая задача у исполнителя закрыта (WIP-лимит)

### В работе → На проверке
- [ ] Файл создан в `output/` (или указано где результат)
- [ ] Все пункты чек-листа отмечены `- [x]`
- [ ] Написан итоговый комментарий: что сделано, что требует внимания Жени
- [ ] Если были отклонения от задачи — зафиксированы явно
- [ ] Критические данные (бюджет ₽, даты, KPI) присутствуют в результате

### На проверке → Готово
- [ ] Женя утвердил ИЛИ Critic дал OK
- [ ] Все замечания из «Не принято» (если были) устранены
- [ ] Финальный файл сохранён в правильной директории
- [ ] Статус в Google Sheets обновлён (если применимо)

### На проверке → Не принято
- [ ] Проверяющий написал конкретный комментарий: что именно не так
- [ ] Указан критерий для повторного принятия: «сделать X → снова на проверку»
- [ ] Исполнитель подтвердил что понял правки

## Приоритизация — матрица Эйзенхауэра

```
                  СРОЧНО          НЕ СРОЧНО
ВАЖНО      │  1. ДЕЛАЙ СЕЙЧАС  │  2. ЗАПЛАНИРУЙ  │
           │  (блокеры, дедл.) │  (тактика, SEO) │
───────────┼───────────────────┼─────────────────┤
НЕ ВАЖНО   │  3. ДЕЛЕГИРУЙ     │  4. УДАЛИ/ОТЛОЖЬ │
           │  (рутина, форматы)│  (хотелки)      │
```

**Квадрат 1 — делаем немедленно:** дедлайн тендера, ответ партнёру с горящим сроком, блокер лайнапа.
**Квадрат 2 — основной фокус:** разделы тактики, контент-план, подготовка питчей.
**Квадрат 3 — делегировать Жене или отложить:** согласовать мелочь, отформатировать готовый документ.
**Квадрат 4 — удалить:** «было бы хорошо», «когда-нибудь».

## Правило «Человек > Техника»

При равном приоритете двух задач — **задача с живым откликом выполняется первой.**

Примеры:
- Ответ амбассадору, который ждёт питч → раньше чем написать ТЗ для Давида
- Follow-up радиостанции после звонка → раньше чем заполнить шаблон таблицы
- Ответ блогеру на запрос о коллаборации → раньше чем оформить отчёт

Логика: живой человек не ждёт. Документ подождёт.

## Правило ковра

**Любая договорённость = карточка в бэклоге.**

Договорились устно / в переписке → немедленно создаём карточку.
Нет карточки = договорённости не существует.

**99% сделано ≠ Готово.**
Задача находится в «Готово» только когда проверяющий явно подтвердил.
«Почти готово», «осталось чуть-чуть», «сделаю сегодня» — задача остаётся «В работе».

## Блокеры

Если задача заблокирована (ждём данные от Жени, фото от Давида, ответ партнёра):

1. **Зафиксировать:** что блокирует, кто должен разблокировать, к какому сроку
2. **Взять следующую задачу из бэклога** (исключение из WIP-лимита)
3. **Если блокер > 2 дней → эскалировать Жене** с конкретным вопросом

Формат фиксации блокера:
```
🔴 БЛОКЕР: [что именно не хватает]
   Ждём от: [Женя / Давид / партнёр]
   Нужно к: [дата]
   Эскалация: [если не получено до дата → сообщить Жене]
```

## Формат отчёта при завершении задачи

```
✅ Задача: [название]
📁 Файл: output/[путь к результату]
📋 Что сделано: [1–3 предложения конкретно]
⚠️  Требует от Жени: [конкретное действие — утвердить / заполнить / проверить]
🔴 Блокеры: [если есть — что именно]
📅 Следующий шаг: [одно действие с дедлайном]
```
