## 0. ПОЧЕМУ v6 И ПОЧЕМУ С НУЛЯ

### Что было не так (19 часов → 0 результата)

1. **Бот = читатель markdown-файлов.** "Покажи анализ рынка" → дампит файл. Агенты живут отдельно от бота.
2. **Агенты = пустые оболочки.** Paperclip зарегистрировал 6 агентов, но у 2 из 6 (CEO, Notifier) нет ни промпта, ни адаптера.
3. **Нет иерархии.** 4 агента-специалиста стоят рядом, а не под руководством. Нет того, кто разбивает задачу, проверяет, координирует.
4. **Бот пытался быть умным.** TOPIC_MAP, ACTION vs QUERY, detectAgentType — 300 строк NLP-роутинга внутри бота. Бот должен быть тупой трубой.
5. **Google Sheets не обновлялись.** Агенты не знали про таблицы. Синхронизация добавлена, но не протестирована в цикле.
6. **Каждая сессия начинала с нуля.** Контекст терялся при переключении моделей (Opus для планирования → Sonnet для выполнения).

### Что должно быть

**Telegram-бот** → **Paperclip (CMO-агент)** → **Иерархия отделов** → **Результаты в TG + Google Sheets**

Один человек (Женя) отправляет голосовое или текст → AI-штаб работает часами/днями → готовые документы, посты, таблицы приходят обратно.

### 1.1. Уровень 0: CMO (Chief Marketing Officer)

**НЕ делает сам:** Не пишет тексты, не ищет данные, не создаёт посты. ТОЛЬКО координирует.

**Входы:** Сообщения из TG через бот, плановые heartbeat (каждые 4 часа)
**Выходы:** Подзадачи для отделов, сводки для TG, обновления в трекинге

- Миссия фестиваля (из context/ExtremeFest_Context.md)

## 2. PAPERCLIP: КАК ЭТО РАБОТАЕТ ТЕХНИЧЕСКИ

### 2.1. Issue как единица работы

Каждая задача в системе = Issue в Paperclip.

```
Issue {
  id: UUID
  identifier: "CMP-42"
  title: "[analyst→researcher] Обнови конкурентный анализ"
  description: "Полное описание задачи..."
  status: backlog | in_progress | done | cancelled
  priority: low | medium | high | urgent
  assigneeAgentId: UUID агента-исполнителя
  // --- Расширения (через comments/description) ---
  parentIssueId: UUID (указывается в description как тег)
  department: strategy | content | growth
  section: а | б | в | ... | н
}
```

### 2.2. Иерархия задач (через naming convention)

Paperclip не имеет встроенных parent/child issues. Реализуем через convention:

```
Пример:
CMP-50: [CMO] Подготовить все 14 разделов тактики
  ├─ CMP-51: [strategy→researcher] Собрать данные для раздела а (стратегия)
  ├─ CMP-52: [strategy→researcher] Собрать данные для раздела б (анализ рынка)
  ├─ CMP-53: [strategy→writer] Написать раздел в (бюджет)
  └─ ...

В description каждого sub-issue:
"PARENT: CMP-50 | SECTION: а | DEPARTMENT: strategy"
```

### 2.3. Flow выполнения задачи

```
1. CMO создаёт Issue:
   POST /api/companies/{id}/issues
   { title: "[strategy→researcher] ...", description: "...", assigneeAgentId: RESEARCHER_ID }

2. CMO будит агента:
   POST /api/agents/{RESEARCHER_ID}/heartbeat/invoke

3. Paperclip запускает agent-runner.js для Researcher:
   - Читает inbox (GET /api/companies/{id}/issues?assigneeAgentId=...)
   - Checkout (POST /api/issues/{id}/checkout)
   - Spawn Claude CLI с промптом
   - Пишет результат как comment
   - PATCH status → done

4. CMO на следующем heartbeat видит, что CMP-51 done:
   - Читает comment с результатом
   - Создаёт следующую задачу: CMP-54 для Writer
   - Цикл повторяется
```

### 2.4. agent-runner.js: Ключевые изменения

**Текущий agent-runner.js** обрабатывает только 1 задачу за запуск. Нужно:

1. **Self-re-invoke:** После завершения задачи — проверить inbox, если есть ещё → POST heartbeat/invoke на себя
2. **Role-specific timeout:** CEO=180s, Researcher=600s, Writer=900s, Critic=300s
3. **Role-specific contextPreamble:** CEO получает API URLs + agent IDs + маппинг разделов. Специалисты получают свои инструкции по Sheets-синку.
4. **Output parsing для CMO:** CMO возвращает JSON-манифест (массив созданных sub-issues), бот его парсит.

## 3. TELEGRAM-БОТ: ТУПАЯ ТРУБА

### 3.1. Принцип

Бот НЕ маршрутизирует. Бот НЕ решает, какому агенту отдать задачу. Бот делает ровно 4 вещи:

1. **Принимает сообщение** (текст, голос, файл)
2. **Создаёт Issue для CMO** в Paperclip
3. **Polling-ит результат** и доставляет в TG
4. **Обрабатывает feedback** (👍 Ок / ✏️ Доработай / ❌ Отмена)

### 3.2. Новый message handler (50 строк вместо 300)

```javascript
bot.on('message', async (msg) => {
  const chatId = msg.chat.id;
  let text = msg.text;

  // 1. Голосовое → расшифровка
  if (msg.voice) {
    text = await transcribeVoice(msg.voice.file_id);
    await bot.sendMessage(chatId, `🎤 ${text.substring(0, 200)}...`);
  }

  if (!text) return;

  // 2. Slash-команды → read-only дашборды (без агентов)
  if (text.startsWith('/')) {
    return handleSlashCommand(chatId, text);
  }

  // 3. ВСЁ остальное → CMO
  await sendToCMO(chatId, text);
});
```

### 3.3. `sendToCMO(chatId, text)` — единая точка входа

```javascript
async function sendToCMO(chatId, text) {
  // 1. Создать Issue для CMO
  const issue = await paperclip.createIssue(
    `[TG] ${text.substring(0, 80)}`,
    `Запрос из Telegram:\n\n${text}\n\nChat ID: ${chatId}`,
    'ceo'  // всегда CMO
  );

  // 2. Статус в TG
  const statusMsg = await bot.sendMessage(chatId, [
    '🧠 <b>CMO принял задачу</b>',
    `📋 <code>${issue.identifier}</code>`,
    '⏳ Анализирую и распределяю...',
  ].join('\n'), { parse_mode: 'HTML' });

  // 3. Разбудить CMO
  await paperclip.invokeHeartbeat(paperclip.AGENT_IDS.ceo);

  // 4. Ждать ответа CMO (3 мин)
  const result = await paperclip.waitForCompletion(issue.id, {
    intervalMs: 5000,
    timeoutMs: 180000,
  });

  // 5. Парсить ответ CMO
  if (result?.status === 'done') {
    const comment = await paperclip.getLatestComment(issue.id);
    const manifest = parseCeoManifest(comment.body);

    if (manifest && manifest.tasks.length > 0) {
      // CMO разбил на подзадачи → запустить batch polling
      await showBatchStatus(chatId, statusMsg, manifest);
      pollBatch(chatId, manifest);
    } else {
      // CMO ответил напрямую (простой запрос)
      await sendLongMessage(chatId, comment.body);
    }
  }
}
```

### 3.4. Slash-команды (read-only, без агентов)

```
/статус    — дашборд проекта (из tracking/*.md)
/контент   — контент-план недели (из Google Sheets)
/бюджет    — бюджет (из tracking/budget.md)
/задачи    — задачи недели (из н-календарный-план.md)
/cjm       — CJM-покрытие (из Google Sheets)
/черновики — посты на одобрение (из Sheets, статус=Черновик)
/помощь    — список команд
/таблица   — ссылки на Google Sheets
/дайджест  — сводка + дедлайны
```

Эти команды НЕ создают Issues. Они читают файлы и Sheets напрямую. Быстро и бесплатно.

### 3.5. Batch Polling — доставка результатов подзадач

Когда CMO создаёт N подзадач, бот показывает прогресс:

```
📦 CMO создал 14 подзадач:

🔍 CMP-51: Стратегия (Researcher) — ⏳
🔍 CMP-52: Анализ рынка (Researcher) — ⏳
✍️ CMP-53: Бюджет (Writer) — ⏳
...

Обновляю каждые 15 сек. Результаты придут по мере готовности.
```

По мере завершения:
```
✅ 3/14: Рекламный бюджет — готово
[Результат: 15,000 символов, отправлен отдельным сообщением]

✅ 4/14: Анализ рынка — готово
[Результат: 8,000 символов]
```

Финально:
```
📊 Все 14 подзадач завершены
✅ Успешно: 13
❌ Ошибок: 1 (CMP-58: таймаут)
⏱ Время: 47 минут

Файлы обновлены:
```

### 4.2. Heartbeat-режим CMO

CMO запускается по heartbeat каждые 4 часа. Даже без запроса из TG, он:

1. **Проверяет inbox** — есть ли новые задачи от бота
2. **Проверяет calendar** — какие дедлайны на этой неделе
3. **Проверяет tracking/** — какие метрики просели
4. **Генерирует задачи** — если что-то просрочено или нужно обновить

Это создаёт ПРОАКТИВНОЕ поведение: CMO сам находит работу для агентов.
